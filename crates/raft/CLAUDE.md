## Purpose

Saga orchestrator, background jobs, apply pipeline, rate limiting, node coordination, leader orchestration. **Despite the name, most of this crate is NOT Raft internals** â€” those live in `consensus`. This crate is the glue between consensus and services: it turns committed proposals into state mutations, runs long-lived background work, and coordinates across nodes.

## Owned Surface

- **Connection pool**: `NodeConnectionRegistry` (`node_registry.rs`) â€” one `tonic::Channel` per peer, HTTP/2-multiplexed across every subsystem.
- **Apply pipeline**: `ApplyPool` + `ApplyWorker` â€” bounded worker pool applying committed proposals to `StateLayer`. Apply-path admin arms call `StateLayer::apply_operations_lazy` (lazy commit); out-of-apply-pipeline admin / recovery callers call `StateLayer::apply_operations` (strict-durable). `OrganizationRequest::IngestExternalEvents` is the apply-path variant for external `IngestEvents` RPC traffic â€” routed through REGIONAL Raft with event IDs frozen pre-consensus, applied via `EventWriter::write_events` (`commit_in_memory`).
- **Handler-phase event batching**: `EventHandle::record_handler_event` enqueues into a bounded `FlushQueue`; a single `EventFlusher` task per `EventHandle` drains on time / size / shutdown trigger and commits via `commit_in_memory` â€” durability is realised by the `StateCheckpointer`'s per-tick events.db sync, the same contract as apply-phase events. `EventHandle::flush_for_shutdown` is invoked from `GracefulShutdown` Phase 5b between WAL flush (5a) and `sync_all_state_dbs` (5c); Phase 5c is what fsyncs the drained batches before exit. `EventWriterBatchConfig::enabled = false` bypasses the queue entirely, routes emissions through `EventWriter::write_entry` directly, and restores per-emission fsync semantics â€” see [`event_writer.rs`](src/event_writer.rs) and `docs/architecture/durability.md` Â§ "Handler-phase event flush window".
- **Transport**: `consensus_transport/` (`mod.rs`, `peer_sender.rs`) â€” backpressure-aware peer messaging for consensus traffic.
- **Saga orchestration**: `SagaOrchestrator` (`saga_orchestrator.rs`).
- **Coordination**: `RaftManager` (`raft_manager.rs`), `ConsensusHandle` (`consensus_handle.rs`), leader orchestration (`leader_lease.rs`, `leader_transfer.rs`).
- **Rate limit + hot-key**: `RateLimiter` (`rate_limit.rs`), `HotKeyDetector` (`hot_key_detector.rs`).
- **Graceful shutdown**: `GracefulShutdown` + `ShutdownCoordinator` (`graceful_shutdown.rs`); `RaftManager::sync_all_state_dbs` invoked from `pre_shutdown` after WAL flush to sync state.db + raft.db per region.
- **Durability lifecycle**: `StateCheckpointer` (`state_checkpointer.rs`) + `RaftLogStore::replay_crash_gap` (`log_storage/raft_impl.rs`) â€” driven by `CheckpointConfig` in `types`; operator reference in `docs/architecture/durability.md`.
- **Snapshot encryption keys**: `SnapshotKeyProvider` trait (`snapshot_key_provider.rs`) â€” symmetric with consensus's `VaultKeyProvider`, but keyed by the residency triple `(region, organization, Option<vault>)` so org-level and per-vault snapshots resolve to independent DEKs. `NoopSnapshotKeyProvider` (always returns `None`; bootstrap default and Stage 2's plaintext-snapshot opt-out) and `InMemorySnapshotKeyProvider` (HashMap behind `RwLock`; tests + single-node) ship in this module. Threaded through `RaftManagerConfig::snapshot_key_provider` -> every `RaftLogStore` (`with_snapshot_key_provider`) -> `LedgerSnapshotBuilder`. Stage 1a stored the provider; Stage 1b stamps the resolved scope into every `LedgerSnapshotBuilder` (org vs vault) and into the LSNP v2 header so Stage 2's `SnapshotPersister` is the first dereferencing call site.
- **Snapshot file format (LSNP v2)**: `crates/raft/src/snapshot.rs` owns the on-disk shape. Stage 1b bumped LSNP from v1 to v2: a `SnapshotScope::{Org, Vault}` marker in the header (encoded in the flags byte plus optional org+vault coords) discriminates org-level snapshots from per-vault snapshots, and the table section is partitioned into typed `TableBlockKind` blocks (`OrgRaft`, `VaultRaft`, `VaultState`, `VaultBlocks`). Org snapshots emit one block (`OrgRaft`); vault snapshots emit three (`VaultRaft`, `VaultState`, `VaultBlocks`) plus the events section. v1 readers are gone â€” files written before the Stage 1b cutover are rejected with `SnapshotError::UnsupportedVersion`. Allowed table-id whitelists per block live in `SNAPSHOT_TABLE_IDS_{ORG_RAFT,VAULT_RAFT,VAULT_STATE,VAULT_BLOCKS}`; `allowed_tables_for_block` is the install-side guard.
- **Background jobs** (crate root): `auto_recovery`, `backup`, `btree_compaction`, `block_compaction`, `post_erasure_compaction`, `integrity_scrubber`, `learner_refresh`, `organization_purge`, `events_gc`, `ttl_gc`, `orphan_cleanup`, `invite_maintenance`, `token_maintenance`, `user_retention`, `dek_rewrap`, `BackgroundJobWatchdog`.
- **Observability**: `metrics.rs`, `otel.rs`, `logging/`, `trace_context.rs`.

## Test Patterns

- Unit tests use in-memory state + `RuntimeConfigHandle` injection for rate-limit / hot-key parameter variation.
- Background job lifecycle tests verify metrics emission + lifecycle log lines â€” not just that the job runs.
- Saga tests use the in-memory state layer; residency is asserted via the `state` crate's proptests.

## Local Golden Rules

1. **Saga PII persists only at `_tmp:saga_pii:{saga_id}`** (Regional + Temporary). PII written to `_meta:saga:` or any GLOBAL key is a residency bug. Enforced by `data-residency-auditor`.
2. **Apply workers never mutate state directly.** They route committed proposals through `StateLayer`. Bypassing the apply pipeline (raw backend writes in a job) corrupts state-machine determinism across replicas.
3. **Graceful shutdown phases are ordered.** The `CancellationToken` propagates through phases 1â†’6 in sequence; reordering (e.g., draining before closing the listener) leaks tasks or drops in-flight requests.
4. **`NodeConnectionRegistry` is the only place to create peer `tonic::Channel`s.** Every subsystem (consensus, discovery, admin, saga) shares channels via this registry â€” HTTP/2 multiplexes them all through one TCP connection per peer. Per-subsystem channels cause connection storms.
5. **Rate limiter + hot-key detector atomic fields are `AtomicU64`.** Non-atomic fields can't be reconfigured via `RuntimeConfigHandle` at runtime.
6. **Client routing is redirect-only.** `RegionalProposal` is the only server-to-server forwarding RPC and exists solely for saga orchestration (root rule 11). Don't extend it.
7. **Service-layer helpers live in `crates/services/src/services/`, not here.** `helpers.rs`, `metadata.rs`, `error_details.rs`, `error_classify.rs` are in `services`. Don't introduce a `src/services/` subdirectory under raft.
8. **`HotKeyDetector` is observational only.** It exposes Prometheus metrics; it does not reject traffic. Adding rejection logic tied to hot-key output is a different RFC.
9. **Background jobs emit lifecycle logs + metrics.** A job that silently runs is a debugging nightmare â€” every job needs `info!("<job> starting")` / `"<job> finished"` + a Prometheus counter.
10. **Deadlines propagate via `deadline.rs`.** Fire-and-forget internal paths are explicit exceptions (documented inline); defaulting to no deadline causes request pileups under load.
11. **`RaftLogStore::replay_crash_gap` runs BEFORE the apply worker is spawned.** A live apply worker during replay would double-apply entries (the WAL range is already in-flight through the replay path). Enforced by `start_region` call ordering in `raft_manager.rs` â€” reordering that site is a correctness bug.
12. **`StateCheckpointer` syncs every per-org and per-vault state.db, raft.db, blocks.db, events.db, and meta.db on every tick.** Raft.db holds `AppliedStateCore` / `applied_durable`; skipping its sync means post-clean-shutdown restarts read `applied_durable = 0` and replay the entire WAL. The per-vault raft.db handles are threaded in via `vault_raft_dbs_fn` (`Stage 5a` prerequisite, landed in commits 820c73c + f07d886) and synced concurrently in Phase A alongside the per-org raft.db, so per-vault `applied_durable_index` advances independently per vault on the checkpoint cadence (~500ms default). Post-crash, `replay_crash_gap` only re-drives `(applied_durable, last_committed]` per vault â€” bounded by checkpoint cadence Ã— per-vault apply rate, not the full WAL since boot. Dormant vaults are filtered out of the fan-out (O6 hibernation Pass 2): every `vault_*_dbs_fn` closure skips Dormant vaults inline, and the per-tick state.db fan-out filters via `dormant_vaults_fn`. Same set-of-DBs invariant holds for `sync_all_state_dbs` in `RaftManager` (graceful shutdown). Stage 5a observability (`ledger_vault_applied_durable_lag` gauge + `ledger_vault_applied_durable_advanced_total` counter + `ledger_vault_applied_durable_advance_duration_seconds` histogram, all emitted from `do_checkpoint`'s success branch) makes the per-vault advance visible to operators. Adding a new per-org or per-vault DB to the durability contract requires updating every checkpoint / shutdown path in lockstep.

13. **Cross-tier communication flows only through typed channels.** `OrganizationGroup` never proposes coordination state; `RegionGroup` never proposes data; `SystemGroup` never proposes regional or per-organization state. The three group types share no apply path and no persistence â€” they communicate exclusively through the typed `CoordinationEvent` / `OrganizationEvent` / `RegionLeaderState` channels (e.g., `organization_creation_rx` on `RaftManager`, the `Delegated` leadership watcher on each per-org `ConsensusState`). Bypassing these channels (region apply directly mutating organization Raft state, or vice versa) is a tier violation and breaks root rule 15. Audited by `consensus-reviewer`.

14. **Per-organization `OrganizationGroup`s and per-vault `VaultGroup`s default to `LeadershipMode::Delegated`.** Election timeouts on their `ConsensusState`s are disabled; per-org leaders follow the parent `RegionGroup`, and per-vault leaders follow the parent `OrganizationGroup` (which in turn follows the region). A watcher task (`start_organization_group` in `raft_manager.rs`) propagates the region leader into each per-org shard via `ConsensusEngine::adopt_leader` â†’ `ConsensusState::adopt_leader`; vault shards adopt the org leader through the same engine. Per-organization self-election (`LeadershipMode::SelfElect`) is a future runtime escape hatch; the data plane (replication, apply, state) is mode-agnostic, so flipping the mode does not require changing the apply path or the state machine. **Per-vault membership cascade flows through `MembershipDispatcher` (`region_membership_watcher.rs`)** â€” `RegionMembershipWatcher` cascades the parent-org conf-change synchronously and enqueues one `MembershipChangeRequest` per child vault into the org's `MembershipQueue`, where the dispatcher drains them under the queue's rate-limit cap (Phase 5 / M3-M5). `RaftManager::cascade_membership_to_children` is preserved as a public API for tests and the legacy escape hatch; no production callers remain.

15. **`RaftManager.regions` is keyed by `(Region, OrganizationId)`; `RaftManager.vault_groups` is keyed by `(Region, OrganizationId, VaultId)`. Every iteration site MUST iterate the full keyspace of whichever map it touches.** Region-control-plane groups live at `(region, OrganizationId(0))`; per-organization groups live at `(region, oid != 0)`. Iterating `list_regions()` (region-only) or filtering on `OrganizationId(0)` skips per-org groups silently â€” a class of bug that has surfaced three times (Tasks #145, #151, #172): startup rehydration, shutdown sync, and shutdown teardown. The same rule applies to `vault_groups`: every per-vault group carries an independent `VaultLifecycleState` (`Active` / `Dormant` / `Stalled`, see `VaultLifecycleState` in `raft_manager.rs`) and a per-shard `MembershipQueue` cursor; iterations that filter on lifecycle state, region, or org without scanning the full `(region, organization_id, vault_id)` keyspace miss vaults silently. Scope-widening a state map is a workspace-wide audit, not a local edit. Detected by review + the consensus-reviewer agent.

16. **`RaftManager::shutdown` tears down groups in reverse order of construction: vaults â†’ per-organization groups â†’ region groups.** Each vault must call `commit_dispatcher.deregister(shard_id)` and `engine.remove_shard(shard_id)` BEFORE the parent org's engine is torn down â€” vaults are children of the org's `ConsensusEngine`, and deregistering after the engine is gone orphans dispatcher entries with no engine to clean up against. Per-organization groups (`org_id != 0`) drop next: cancel background jobs, request reactor shutdown, drop transport (which drops `PeerSender`s and cancels drain tasks via `Drop`, closing in-flight `Replicate` streams). Region groups (`org_id == 0`) drop last via `stop_region`. Inverting any pair leaves H2 streams open on the shared `NodeConnectionRegistry` channel and causes subsequent `client.replicate()` calls on a `graceful_restart` to hang in partial-handshake state. Phases are explicit at `RaftManager::shutdown` in `raft_manager.rs`; if a fourth tier is added, slot it before the existing phase 1 â€” never after the region phase. Surfaced by Task #172.

17. **One `ConsensusEngine` (and one `Reactor`, one `GrpcConsensusTransport`) per `(Region, OrganizationId)` tuple â€” vault shards share the parent org's engine.** Vault shards are added via `engine.add_shard` on the parent org's `ConsensusEngine`; they do not get their own engines, reactors, or transports. Each vault group does get its own `ConsensusHandle` via `ConsensusHandle::new_for_shard` (so `vault.handle().propose(...)` routes to the vault's `shard_id`, not the org's), its own per-shard `RaftLogStore` with per-vault `block_archive`, its own cancellation token, and its own `VaultLifecycleState` + `last_activity_at` AtomicU64 (touched on every proposal; the idle-detector reads them per-tick). Vault leader is adopted from the parent org's leader via `Shard::adopt_leader` â€” vaults never elect independently. Adding an engine per vault, or breaking the shared transport, would multiply H2 stream counts and break the symmetric-gap audits in rule 15. Engine ownership lives on `InnerGroup` in `raft_manager.rs`; vault membership lives on `InnerVaultGroup`. **O6 hibernation Pass 2 has landed: raft-layer wake/sleep wiring + page-cache eviction.** `RaftManager::sleep_vault(region, org, vault)` is called from the hibernation idle detector's Active â†’ Dormant arm (dispatched as a tokio task so the sync scan loop is not blocked); it pauses the vault's consensus shard via `engine.pause_shard(shard_id)`, transitions the vault to `VaultLifecycleState::Dormant`, and best-effort evicts the OS page cache for the four per-vault DBs (`state.db` via `StateLayer::evict_vault_state_page_cache`; `raft.db`, `blocks.db`, `events.db` via `Database::evict_page_cache` directly). `RaftManager::wake_vault(...)` resumes the shard via `engine.resume_shard(shard_id)` and transitions back to Active; page-cache repopulation is **lazy** â€” the next read after wake re-fetches pages from disk on Linux (no explicit reopen). The wake notifier (`RaftManagerWakeNotifier`) carries an `Arc<Mutex<Weak<RaftManager>>>` shared with `RaftManager::self_weak`; bootstrap fills it via `RaftManager::install_self_weak()` immediately after wrapping the manager in `Arc`. The notifier translates `ShardWakeNotifier::on_peer_message_for_paused_shard(shard_id)` into a `tokio::spawn(wake_vault(triple))` via the reverse `vault_shard_index: DashMap<ConsensusStateId, (Region, OrganizationId, VaultId)>` maintained alongside `vault_groups` inserts/removes â€” drop-and-let-Raft-retry: the peer's next AppendEntries heartbeat (~50ms) lands on the awoken shard. **`Arc<Database>` clones on `InnerVaultGroup` stay alive across sleep/wake** â€” FDs are not closed; the operator-visible win is memory pressure (Linux `posix_fadvise(POSIX_FADV_DONTNEED)` drops cached pages), not FD count. Page-cache eviction is best-effort: macOS / Windows return `Ok(())` no-op (no equivalent to `posix_fadvise` without an mmap roundtrip; see `crates/fs/CLAUDE.md`). **The `StateCheckpointer` skips Dormant vaults entirely** â€” every per-tick fan-out (state.db via filtered `live_vault_dbs()`, raft.db / blocks.db / events.db via the `vault_*_dbs_fn` closures) filters on `inner.lifecycle_state() != Dormant` so re-syncing does not refault the OS pages we just dropped. Dormant vaults have no in-flight writes (apply pipeline drains at the Active â†’ Dormant boundary), so the skipped sync is also a no-op early-return inside `Database::sync_state` â€” both paths land at the same outcome. **Graceful shutdown** (`RaftManager::sync_all_state_dbs`) does NOT explicitly skip Dormant vaults; their `last_synced_snapshot_id == committed.snapshot_id` invariant means the sync_state coalesce check returns Ok early without an fsync, so shutdown is correct without lifecycle awareness. Wake p99 budget: < 100ms â€” validated in-process by `test_wake_latency_p99_budget` in `raft_manager.rs`. The full subprocess stress test (`crates/server/tests/hibernation_overhead.rs`) is Pass 3 follow-up.

18. **Snapshots are scoped (`SnapshotScope::Org` or `SnapshotScope::Vault { vault_id }`); cross-scope installs are runtime errors.** `LedgerSnapshotBuilder` and `RaftLogStore::install_snapshot` (`log_storage/raft_impl.rs`) both dispatch on the scope marker in the LSNP v2 header. Org-scoped builders read only the org's `raft.db`; vault-scoped builders read the vault's `raft.db` + `state.db` (via `StateLayer::db_for(vault_id)`) + `blocks.db` (via `BlockArchive::db()`) + the apply-phase event stream. `install_snapshot` rejects (a) an org snapshot installed into a per-vault store, (b) a vault snapshot installed into an org store, and (c) a vault A snapshot installed into a vault B store, all with `SnapshotError::ScopeMismatch` BEFORE any data is written. Writing entity / block tables into the org-level `raft.db` (or org-level tables into a per-vault DB) is the structural correctness bug Stage 1b fixes; per-block-kind table-id whitelists in `crates/raft/src/snapshot.rs` are defense-in-depth. **Stage 2 wires persistence**: `SnapshotPersister` (`snapshot_persister.rs`) consumes the builder's plaintext file, applies a whole-file AES-256-GCM envelope via `inferadb_ledger_consensus::snapshot_crypto::encrypt_snapshot` (skipped when the `SnapshotKeyProvider` returns `None` â€” only valid for tests / unencrypted local-dev configurations), atomic-write-temp-renames into `{snapshot_dir}/snapshot-{index:020}.snap` (org) or `{snapshot_dir}/vault-{vault_id}/snapshot-{index:020}.snap` (vault), and prunes older snapshots beyond `retention_count` (default `3`). Retention failures are non-fatal â€” the persister logs a warn and increments `ledger_snapshot_retention_failed_total` but does not surface as a persist error since the new snapshot is already durable. Receiver-side install (Stage 4) decrypts the envelope first, then runs `SnapshotReader::verify_checksum` on the recovered plaintext. **Stage 3 wires wire transfer**: streamed-from-leader files land at `{snapshot_dir}/staging-{index:020}.snap.staged` (or under the `vault-{vault_id}/` subdir for vault scopes) â€” the `.snap.staged` suffix keeps streamed files distinct from locally persisted `*.snap` files so `SnapshotPersister::list` (Rule 18 enumeration) and `SnapshotPersister::list_staged` (Rule 21 enumeration) return disjoint sets. Receiver-side validation (CRC32 + final byte count) lives in `crates/raft/src/snapshot_receiver.rs`; decryption + LSNP-verify + Raft install remain Stage 4's job.

19. **`MembershipQueue` is bounded; per-vault conf-changes have a timeout and a stalled-state metric.** Each per-organization `InnerGroup` carries an `Option<Arc<MembershipQueue>>` (populated only for `organization_id != OrganizationId(0)`; the system group and the regional control-plane group leave it `None`). Default capacity: 2 concurrent in-flight (matching TiKV's `max_snapshot_per_store`), 1000 backlog. Every entry the `MembershipDispatcher` pops is wrapped in `tokio::time::timeout(vault_conf_change_timeout_secs)` (default 60s). On timeout the dispatcher logs at `warn`, increments `ledger_vault_conf_change_stalled_total{region, organization_id, vault_id, action, reason=timeout}`, and drops the entry â€” the cascade is best-effort, and the next region-state delta `RegionMembershipWatcher` observes will re-derive the change. Operators on slow networks can raise `vault_conf_change_timeout_secs` via `RaftManagerConfig`; lowering it makes a truly broken vault visible faster but trades against legitimate slow-Raft scenarios. Audited by `consensus-reviewer`.

20. **`Action::TriggerSnapshot` dispatches off the reactor task via `RaftManagerSnapshotCoordinator`.** Stage 2 replaces the no-op `Action::TriggerSnapshot` arm in `crates/consensus/src/reactor.rs` with a callback into a `SnapshotCoordinator` trait (mirror of `ShardWakeNotifier`). The raft crate's `RaftManagerSnapshotCoordinator` holds the same `Arc<Mutex<Weak<RaftManager>>>` cell as the wake notifier (filled by `RaftManager::install_self_weak`), `tokio::spawn`s the build + persist + notify chain, and resolves the shard's scope via two reverse maps maintained alongside `regions` / `vault_groups`: `org_shard_index: DashMap<ConsensusStateId, (Region, OrganizationId)>` for org-level shards, `vault_shard_index` for vault shards. Both maps must be updated in lockstep with `regions.insert` / `vault_groups.insert` â€” a missing `org_shard_index` entry surfaces as a `tracing::warn!("no scope mapping for trigger; dropping")` and the next threshold check retries; a stale entry routes the build into a torn-down group. **`RaftManager::persist_snapshot`** is the first production caller of `ConsensusEngine::notify_snapshot_completed` â€” it builds the snapshot (via a `LedgerSnapshotBuilder` constructed off `InnerGroup` / `InnerVaultGroup` Arc handles), persists via the shared `SnapshotPersister` (Rule 18), then notifies the engine so the shard's `last_snapshot_index` advances. Wired into `ConsensusEngine::start_with_full_coordinators` from `RaftManager::start_region` (alongside `RaftManagerWakeNotifier` and `RaftManagerSnapshotSender`). Audited by `consensus-reviewer`.

21. **`Action::SendSnapshot` dispatches off the reactor task via `RaftManagerSnapshotSender`.** Stage 3 replaces the empty-`Vec::new()` `Message::InstallSnapshot` placeholder in `crates/consensus/src/reactor.rs` with a callback into a `SnapshotSender` trait (mirror of `SnapshotCoordinator`). The raft crate's `RaftManagerSnapshotSender` (`crates/raft/src/snapshot_streamer.rs`) holds the same `Arc<Mutex<Weak<RaftManager>>>` cell as the wake notifier and snapshot coordinator (filled by `RaftManager::install_self_weak`), `tokio::spawn`s the streaming-send chain, and resolves the shard's scope via the same `org_shard_index` / `vault_shard_index` reverse maps used by Stage 2 (Rule 20). The dispatched task: (1) opens the at-rest encrypted snapshot via `SnapshotPersister::open_encrypted` (raw envelope bytes â€” no decrypt-then-re-encrypt), (2) connects to the follower via the shared `peer_addresses` map, (3) ships chunks (default 256 KiB) over the internal Raft transport's `InstallSnapshotStream` RPC. The wire format mirrors the at-rest format verbatim â€” on-the-wire encryption is provided by WireGuard at the link layer, so the streaming protocol carries the encrypted envelope as opaque bytes. Failures (no scope mapping, follower unreachable, RPC error, short read) log + record `ledger_snapshot_send_total{result=failure}` and drop the request; the leader's heartbeat replicator re-emits `Action::SendSnapshot` on the next cycle, so transient failures self-heal without explicit retry state. **Receiver-side**: `crates/raft/src/snapshot_receiver.rs::receive_install_snapshot_stream` validates header â†’ data â†’ footer ordering, computes a CRC32 over the data chunks, and writes the encrypted bytes to a per-scope staging file (`staging-{index:020}.snap.staged`) under the same directory `SnapshotPersister` uses. Stage 4's installer discovers staged files via `SnapshotPersister::list_staged` and consumes them through `RaftLogStore::install_snapshot`. Wired into the gRPC `RaftService` impl in `crates/services/src/services/raft.rs`. Audited by `consensus-reviewer`.

22. **`Action::InstallSnapshot` dispatches off the reactor task via `RaftManagerSnapshotInstaller`.** Stage 4 replaces the no-op receiver-side `Message::InstallSnapshot` handler in `crates/consensus/src/consensus_state.rs` with a `SnapshotInstaller` trait callback (sister of `SnapshotSender`). The receiver-side handler now emits `Action::InstallSnapshot { shard, leader_term, last_included_index, last_included_term }` whenever `last_included_index > last_snapshot_index` (idempotent â€” covered indices skip the dispatch); the reactor routes the action into the registered installer. The raft crate's `RaftManagerSnapshotInstaller` (`crates/raft/src/snapshot_installer.rs`) holds the same `Arc<Mutex<Weak<RaftManager>>>` cell as the wake notifier / snapshot coordinator / snapshot sender (filled by `RaftManager::install_self_weak`), `tokio::spawn`s the install chain, and resolves the shard's scope via `RaftManager::resolve_shard_to_scope` (reusing the Stage 2 `org_shard_index` + `vault_shard_index` reverse maps). The dispatched task: (1) polls `SnapshotPersister::list_staged` up to `DEFAULT_STAGED_WAIT` (5s default) waiting for the file to land â€” covers the race where `Action::InstallSnapshot` arrives before the streaming RPC's atomic rename completes; (2) reads the file and decrypts via the per-scope `SnapshotKeyProvider` / `decrypt_snapshot` (decrypt failure is unrecoverable for this follower â€” the staged file is pruned and the failure surfaces through `ledger_snapshot_install_decrypt_failed_total{scope_kind}`); (3) writes the recovered LSNP-v2 plaintext to a temp `tokio::fs::File` and routes an `ApplyCommand::InstallSnapshot { meta, plaintext, completion }` onto the per-shard `apply_command::ApplyCommandSender` stored on the apply task that owns the shard's `RaftLogStore` â€” `InnerGroup::apply_command_tx` for org-level scopes (drained by `ApplyWorker::run` via `tokio::select!` alongside the commit channel) and `InnerVaultGroup::apply_command_tx` for per-vault scopes (drained by the inlined commit pump in `RaftManager::start_vault_group` via the same `tokio::select!` shape). Strict ordering on both pumps: under `biased;` selection the install arm wins over batch arrivals, and the install `await`s `RaftLogStore::install_snapshot` to completion before yielding back to the loop, so concurrent batch apply during the wholesale state-replace is structurally impossible. (4) the apply task calls `RaftLogStore::install_snapshot` (Stage 1b's bifurcated install with cross-scope rejection) and forwards the result on the oneshot; (5) on success the installer calls `ConsensusEngine::notify_snapshot_installed(shard, idx, term)` so the shard advances `last_applied` / `last_snapshot_index` / `commit_index` and truncates the log prefix subsumed by the snapshot, then prunes the staged file. Failures drop the request; the leader's heartbeat replicator re-emits `Action::SendSnapshot` on the next cycle (restaging the file via Stage 3 and re-emitting this action), so transient failures self-heal without explicit retry state. Metrics: `ledger_snapshot_install_total{scope_kind, result}` with `result âˆˆ {success, failure, staged_not_found, decrypt_failed, scope_mismatch}`, plus `ledger_snapshot_install_duration_seconds` and `ledger_snapshot_install_bytes` histograms. End-to-end coverage: `crates/server/tests/snapshot_install.rs` exercises the full chain `persist_snapshot` â†’ reactor dispatch (via the public test helpers `RaftManager::send_snapshot_for_test` and `install_snapshot_for_test`, which invoke the same `SnapshotSender` / `SnapshotInstaller` implementations the reactor uses) â†’ streamer â†’ receiver â†’ installer for both org and vault scopes against a real 3-node TCP cluster. 23. **Stage 5b â€” `RaftManager::start_region` drives parallel per-vault catch-up from the local per-org WAL on restart.** Per-vault `applied_durable_index` lives in the page cache between `StateCheckpointer` ticks (~500ms default cadence); an unclean shutdown loses anything not yet synced to the per-vault `raft.db` dual-slot. The org-level `RaftLogStore::replay_crash_gap` only re-drives the org shard, so without Stage 5b every vault's pre-crash committed-but-unsynced applies are lost on a single-node restart (the live commit pump that runs once `ConsensusEngine` starts only re-applies entries that arrive AFTER startup; on a single-node restart there are no peers to push them, and the entries sit in the WAL forever â€” verified by the `restart_loses_pre_crash_per_vault_commits_without_replay` test in `crates/raft/src/log_storage/parallel_replay.rs`). `RaftManager::run_stage5b_parallel_replay` (in `raft_manager.rs`) is invoked from `start_region` AFTER `replay_crash_gap` and BEFORE `ConsensusEngine::start_with_all_callbacks` consumes the WAL by value. It opens each child vault's store via the shared `RaftManager::open_and_wire_vault_store` helper (the same wiring `start_vault_group` uses â€” load-bearing because `start_vault_group` reopens the same on-disk path later when the org apply pipeline drives vault creations), reads each vault's `applied_durable_index`, runs the M4 primitive `replay_shared_wal_for_org` (`crates/raft/src/log_storage/parallel_replay.rs`) to fan apply across the gap, then drops the stores. **The apply closure MUST `sync_state` per-vault `raft.db` / `blocks.db` / `events.db` before returning the store to its slot** â€” `apply_committed_entries` writes via `commit_in_memory`, which lives in the per-`Database` in-process page cache; when the per-vault `RaftLogStore` (and the wrapping `BlockArchive` / `EventWriter`) drop at the end of the replay, that cache drops with them. Without the explicit sync, the replay is effectively a no-op: `start_vault_group`'s subsequent reopen reads the dual-slot from disk (pre-Stage-5b state). Per-vault `state.db` is owned by the org's long-lived `StateLayer` (it stays in `live_vault_dbs()` after the materialise that `apply_request_with_events` triggered); the first `StateCheckpointer` tick after engine startup picks it up via the standard fan-out, so the closure does not sync state.db explicitly. Skip arms: `enable_parallel_wal_replay = false` (`SKIPPED_DISABLED`), org has zero vaults (no metric â€” cold-start no-op), every vault `applied_durable >= last_committed` (`SKIPPED_NO_GAP`, the clean-shutdown path). Errors are non-fatal and fall through to the live commit pump (`ERROR` outcome label) so an unclean-shutdown gap that fails to close is degraded but not blocking. **Helper extraction is load-bearing**: `open_and_wire_vault_store` is the single source of truth for per-vault store wiring (state_layer / block_archive / event_writer / leader_lease / snapshot_key_provider). Both `start_region`'s replay path and `start_vault_group`'s production-startup path call into it; any divergence between the two would cause the post-replay reopen to observe a different component-handle topology than the replay itself ran against. Metrics: `ledger_org_parallel_replay_invocations_total{region, organization_id, result}` (always emitted; `result âˆˆ {success, error, skipped_no_gap, skipped_disabled}`), plus `ledger_org_parallel_replay_duration_seconds` (histogram, success-only) / `ledger_org_parallel_replay_entries_processed_total` (counter, success-only) / `ledger_org_parallel_replay_vaults_active` (gauge, success-only).

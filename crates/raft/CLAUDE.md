## Purpose

Saga orchestrator, background jobs, apply pipeline, rate limiting, node coordination, leader orchestration. **Despite the name, most of this crate is NOT Raft internals** â€” those live in `consensus`. This crate is the glue between consensus and services: it turns committed proposals into state mutations, runs long-lived background work, and coordinates across nodes.

## Owned Surface

- **Connection pool**: `NodeConnectionRegistry` (`node_registry.rs`) â€” one `tonic::Channel` per peer, HTTP/2-multiplexed across every subsystem.
- **Apply pipeline**: `ApplyPool` + `ApplyWorker` â€” bounded worker pool applying committed proposals to `StateLayer`. Apply-path admin arms call `StateLayer::apply_operations_lazy` (lazy commit); out-of-apply-pipeline admin / recovery callers call `StateLayer::apply_operations` (strict-durable). `OrganizationRequest::IngestExternalEvents` is the apply-path variant for external `IngestEvents` RPC traffic â€” routed through REGIONAL Raft with event IDs frozen pre-consensus, applied via `EventWriter::write_events` (`commit_in_memory`).
- **Handler-phase event batching**: `EventHandle::record_handler_event` enqueues into a bounded `FlushQueue`; a single `EventFlusher` task per `EventHandle` drains on time / size / shutdown trigger and commits via `commit_in_memory` â€” durability is realised by the `StateCheckpointer`'s per-tick events.db sync, the same contract as apply-phase events. `EventHandle::flush_for_shutdown` is invoked from `GracefulShutdown` Phase 5b between WAL flush (5a) and `sync_all_state_dbs` (5c); Phase 5c is what fsyncs the drained batches before exit. `EventWriterBatchConfig::enabled = false` bypasses the queue entirely, routes emissions through `EventWriter::write_entry` directly, and restores per-emission fsync semantics â€” see [`event_writer.rs`](src/event_writer.rs) and `docs/architecture/durability.md` Â§ "Handler-phase event flush window".
- **Transport**: `consensus_transport/` (`mod.rs`, `peer_sender.rs`) â€” backpressure-aware peer messaging for consensus traffic.
- **Saga orchestration**: `SagaOrchestrator` (`saga_orchestrator.rs`).
- **Coordination**: `RaftManager` (`raft_manager.rs`), `ConsensusHandle` (`consensus_handle.rs`), leader orchestration (`leader_lease.rs`, `leader_transfer.rs`).
- **Rate limit + hot-key**: `RateLimiter` (`rate_limit.rs`), `HotKeyDetector` (`hot_key_detector.rs`).
- **Graceful shutdown**: `GracefulShutdown` + `ShutdownCoordinator` (`graceful_shutdown.rs`); `RaftManager::sync_all_state_dbs` invoked from `pre_shutdown` after WAL flush to sync state.db + raft.db per region.
- **Durability lifecycle**: `StateCheckpointer` (`state_checkpointer.rs`) + `RaftLogStore::replay_crash_gap` (`log_storage/raft_impl.rs`) â€” driven by `CheckpointConfig` in `types`; operator reference in `docs/architecture/durability.md`.
- **Background jobs** (crate root): `auto_recovery`, `backup`, `btree_compaction`, `block_compaction`, `post_erasure_compaction`, `integrity_scrubber`, `learner_refresh`, `organization_purge`, `events_gc`, `ttl_gc`, `orphan_cleanup`, `invite_maintenance`, `token_maintenance`, `user_retention`, `dek_rewrap`, `BackgroundJobWatchdog`.
- **Observability**: `metrics.rs`, `otel.rs`, `dogstatsd.rs`, `logging/`, `trace_context.rs`.

## Test Patterns

- Unit tests use in-memory state + `RuntimeConfigHandle` injection for rate-limit / hot-key parameter variation.
- Background job lifecycle tests verify metrics emission + lifecycle log lines â€” not just that the job runs.
- Saga tests use the in-memory state layer; residency is asserted via the `state` crate's proptests.

## Local Golden Rules

1. **Saga PII persists only at `_tmp:saga_pii:{saga_id}`** (Regional + Temporary). PII written to `_meta:saga:` or any GLOBAL key is a residency bug. Enforced by `data-residency-auditor`.
2. **Apply workers never mutate state directly.** They route committed proposals through `StateLayer`. Bypassing the apply pipeline (raw backend writes in a job) corrupts state-machine determinism across replicas.
3. **Graceful shutdown phases are ordered.** The `CancellationToken` propagates through phases 1â†’6 in sequence; reordering (e.g., draining before closing the listener) leaks tasks or drops in-flight requests.
4. **`NodeConnectionRegistry` is the only place to create peer `tonic::Channel`s.** Every subsystem (consensus, discovery, admin, saga) shares channels via this registry â€” HTTP/2 multiplexes them all through one TCP connection per peer. Per-subsystem channels cause connection storms.
5. **Rate limiter + hot-key detector atomic fields are `AtomicU64`.** Non-atomic fields can't be reconfigured via `RuntimeConfigHandle` at runtime.
6. **Client routing is redirect-only.** `RegionalProposal` is the only server-to-server forwarding RPC and exists solely for saga orchestration (root rule 11). Don't extend it.
7. **Service-layer helpers live in `crates/services/src/services/`, not here.** `helpers.rs`, `metadata.rs`, `error_details.rs`, `error_classify.rs` are in `services`. Don't introduce a `src/services/` subdirectory under raft.
8. **`HotKeyDetector` is observational only.** It exposes Prometheus metrics; it does not reject traffic. Adding rejection logic tied to hot-key output is a different RFC.
9. **Background jobs emit lifecycle logs + metrics.** A job that silently runs is a debugging nightmare â€” every job needs `info!("<job> starting")` / `"<job> finished"` + a Prometheus counter.
10. **Deadlines propagate via `deadline.rs`.** Fire-and-forget internal paths are explicit exceptions (documented inline); defaulting to no deadline causes request pileups under load.
11. **`RaftLogStore::replay_crash_gap` runs BEFORE the apply worker is spawned.** A live apply worker during replay would double-apply entries (the WAL range is already in-flight through the replay path). Enforced by `start_region` call ordering in `raft_manager.rs` â€” reordering that site is a correctness bug.
12. **`StateCheckpointer` syncs BOTH state.db and raft.db on every tick.** Raft.db holds `AppliedStateCore` / `applied_durable`; skipping its sync means post-clean-shutdown restarts read `applied_durable = 0` and replay the entire WAL. Same invariant holds for `sync_all_state_dbs` in `RaftManager`. Both sync targets are enumerated in the code; adding a third DB to the durability contract requires updating every checkpoint / shutdown path in lockstep.

13. **Cross-tier communication flows only through typed channels.** `OrganizationGroup` never proposes coordination state; `RegionGroup` never proposes data; `SystemGroup` never proposes regional or per-organization state. The three group types share no apply path and no persistence â€” they communicate exclusively through the typed `CoordinationEvent` / `OrganizationEvent` / `RegionLeaderState` channels (e.g., `organization_creation_rx` on `RaftManager`, the `Delegated` leadership watcher on each per-org `ConsensusState`). Bypassing these channels (region apply directly mutating organization Raft state, or vice versa) is a tier violation and breaks root rule 15. Audited by `consensus-reviewer`.

14. **Per-organization `OrganizationGroup`s and per-vault `VaultGroup`s default to `LeadershipMode::Delegated`.** Election timeouts on their `ConsensusState`s are disabled; per-org leaders follow the parent `RegionGroup`, and per-vault leaders follow the parent `OrganizationGroup` (which in turn follows the region). A watcher task (`start_organization_group` in `raft_manager.rs`) propagates the region leader into each per-org shard via `ConsensusEngine::adopt_leader` â†’ `ConsensusState::adopt_leader`; vault shards adopt the org leader through the same engine. Per-organization self-election (`LeadershipMode::SelfElect`) is a future runtime escape hatch; the data plane (replication, apply, state) is mode-agnostic, so flipping the mode does not require changing the apply path or the state machine. **Per-vault membership cascade flows through `MembershipDispatcher` (`region_membership_watcher.rs`)** â€” `RegionMembershipWatcher` cascades the parent-org conf-change synchronously and enqueues one `MembershipChangeRequest` per child vault into the org's `MembershipQueue`, where the dispatcher drains them under the queue's rate-limit cap (Phase 5 / M3-M5). `RaftManager::cascade_membership_to_children` is preserved as a public API for tests and the legacy escape hatch; no production callers remain.

15. **`RaftManager.regions` is keyed by `(Region, OrganizationId)`; `RaftManager.vault_groups` is keyed by `(Region, OrganizationId, VaultId)`. Every iteration site MUST iterate the full keyspace of whichever map it touches.** Region-control-plane groups live at `(region, OrganizationId(0))`; per-organization groups live at `(region, oid != 0)`. Iterating `list_regions()` (region-only) or filtering on `OrganizationId(0)` skips per-org groups silently â€” a class of bug that has surfaced three times (Tasks #145, #151, #172): startup rehydration, shutdown sync, and shutdown teardown. The same rule applies to `vault_groups`: every per-vault group carries an independent `VaultLifecycleState` (`Active` / `Dormant` / `Stalled`, see `VaultLifecycleState` in `raft_manager.rs`) and a per-shard `MembershipQueue` cursor; iterations that filter on lifecycle state, region, or org without scanning the full `(region, organization_id, vault_id)` keyspace miss vaults silently. Scope-widening a state map is a workspace-wide audit, not a local edit. Detected by review + the consensus-reviewer agent.

16. **`RaftManager::shutdown` tears down groups in reverse order of construction: vaults â†’ per-organization groups â†’ region groups.** Each vault must call `commit_dispatcher.deregister(shard_id)` and `engine.remove_shard(shard_id)` BEFORE the parent org's engine is torn down â€” vaults are children of the org's `ConsensusEngine`, and deregistering after the engine is gone orphans dispatcher entries with no engine to clean up against. Per-organization groups (`org_id != 0`) drop next: cancel background jobs, request reactor shutdown, drop transport (which drops `PeerSender`s and cancels drain tasks via `Drop`, closing in-flight `Replicate` streams). Region groups (`org_id == 0`) drop last via `stop_region`. Inverting any pair leaves H2 streams open on the shared `NodeConnectionRegistry` channel and causes subsequent `client.replicate()` calls on a `graceful_restart` to hang in partial-handshake state. Phases are explicit at `RaftManager::shutdown` in `raft_manager.rs`; if a fourth tier is added, slot it before the existing phase 1 â€” never after the region phase. Surfaced by Task #172.

17. **One `ConsensusEngine` (and one `Reactor`, one `GrpcConsensusTransport`) per `(Region, OrganizationId)` tuple â€” vault shards share the parent org's engine.** Vault shards are added via `engine.add_shard` on the parent org's `ConsensusEngine`; they do not get their own engines, reactors, or transports. Each vault group does get its own `ConsensusHandle` via `ConsensusHandle::new_for_shard` (so `vault.handle().propose(...)` routes to the vault's `shard_id`, not the org's), its own per-shard `RaftLogStore` with per-vault `block_archive`, its own cancellation token, and its own `VaultLifecycleState` + `last_activity_at` AtomicU64 (touched on every proposal; the idle-detector reads them per-tick). Vault leader is adopted from the parent org's leader via `Shard::adopt_leader` â€” vaults never elect independently. Adding an engine per vault, or breaking the shared transport, would multiply H2 stream counts and break the symmetric-gap audits in rule 15. Engine ownership lives on `InnerGroup` in `raft_manager.rs`; vault membership lives on `InnerVaultGroup`. **O6 hibernation Pass 2 has landed: raft-layer wake/sleep wiring + page-cache eviction.** `RaftManager::sleep_vault(region, org, vault)` is called from the hibernation idle detector's Active â†’ Dormant arm (dispatched as a tokio task so the sync scan loop is not blocked); it pauses the vault's consensus shard via `engine.pause_shard(shard_id)`, transitions the vault to `VaultLifecycleState::Dormant`, and best-effort evicts the OS page cache for the four per-vault DBs (`state.db` via `StateLayer::evict_vault_state_page_cache`; `raft.db`, `blocks.db`, `events.db` via `Database::evict_page_cache` directly). `RaftManager::wake_vault(...)` resumes the shard via `engine.resume_shard(shard_id)` and transitions back to Active; page-cache repopulation is **lazy** â€” the next read after wake re-fetches pages from disk on Linux (no explicit reopen). The wake notifier (`RaftManagerWakeNotifier`) carries an `Arc<Mutex<Weak<RaftManager>>>` shared with `RaftManager::self_weak`; bootstrap fills it via `RaftManager::install_self_weak()` immediately after wrapping the manager in `Arc`. The notifier translates `ShardWakeNotifier::on_peer_message_for_paused_shard(shard_id)` into a `tokio::spawn(wake_vault(triple))` via the reverse `vault_shard_index: DashMap<ConsensusStateId, (Region, OrganizationId, VaultId)>` maintained alongside `vault_groups` inserts/removes â€” drop-and-let-Raft-retry: the peer's next AppendEntries heartbeat (~50ms) lands on the awoken shard. **`Arc<Database>` clones on `InnerVaultGroup` stay alive across sleep/wake** â€” FDs are not closed; the operator-visible win is memory pressure (Linux `posix_fadvise(POSIX_FADV_DONTNEED)` drops cached pages), not FD count. Page-cache eviction is best-effort: macOS / Windows return `Ok(())` no-op (no equivalent to `posix_fadvise` without an mmap roundtrip; see `crates/fs-sync/CLAUDE.md`). **The `StateCheckpointer` skips Dormant vaults entirely** â€” every per-tick fan-out (state.db via filtered `live_vault_dbs()`, raft.db / blocks.db / events.db via the `vault_*_dbs_fn` closures) filters on `inner.lifecycle_state() != Dormant` so re-syncing does not refault the OS pages we just dropped. Dormant vaults have no in-flight writes (apply pipeline drains at the Active â†’ Dormant boundary), so the skipped sync is also a no-op early-return inside `Database::sync_state` â€” both paths land at the same outcome. **Graceful shutdown** (`RaftManager::sync_all_state_dbs`) does NOT explicitly skip Dormant vaults; their `last_synced_snapshot_id == committed.snapshot_id` invariant means the sync_state coalesce check returns Ok early without an fsync, so shutdown is correct without lifecycle awareness. Wake p99 budget: < 100ms â€” validated in-process by `test_wake_latency_p99_budget` in `raft_manager.rs`. The full subprocess stress test (`crates/server/tests/hibernation_overhead.rs`) is Pass 3 follow-up.

18. **`MembershipQueue` is bounded; per-vault conf-changes have a timeout and a stalled-state metric.** Each per-organization `InnerGroup` carries an `Option<Arc<MembershipQueue>>` (populated only for `organization_id != OrganizationId(0)`; the system group and the regional control-plane group leave it `None`). Default capacity: 2 concurrent in-flight (matching TiKV's `max_snapshot_per_store`), 1000 backlog. Every entry the `MembershipDispatcher` pops is wrapped in `tokio::time::timeout(vault_conf_change_timeout_secs)` (default 60s). On timeout the dispatcher logs at `warn`, increments `ledger_vault_conf_change_stalled_total{region, organization_id, vault_id, action, reason=timeout}`, and drops the entry â€” the cascade is best-effort, and the next region-state delta `RegionMembershipWatcher` observes will re-derive the change. Operators on slow networks can raise `vault_conf_change_timeout_secs` via `RaftManagerConfig`; lowering it makes a truly broken vault visible faster but trades against legitimate slow-Raft scenarios. Audited by `consensus-reviewer`.
